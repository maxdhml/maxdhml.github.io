## Ce que j'ai fait

J'ai sélectionné, modélisé et exploité différents mécanismes de gestion de données (bases de données relationnelles et fichiers plats) en fonction des besoins techniques des projets.

* **Bases de données relationnelles (R2.07 / R2.09) :** J'ai conçu des modèles conceptuels de données (MCD) et je les ai traduits en tables. J'ai utilisé le langage SQL pour insérer, mettre à jour et interroger des données structurées de manière sécurisée (prévention des injections SQL lors du développement web).
* **Traitement de fichiers plats et Open Data (R2.08 / SAÉ 1.05) :** Pour le projet "Agriculture Pêche", plutôt que de construire une base de données complexe, j'ai choisi de manipuler directement les flux CSV fournis par FranceAgriMer. J'ai utilisé des scripts Python (avec la bibliothèque `pandas`) pour lire, nettoyer et croiser ces gros volumes de données à la volée.

## Pourquoi

En informatique, il n'existe pas de solution de stockage unique : il faut choisir le mécanisme adapté au besoin. Une base de données relationnelle (SGBD) est indispensable pour gérer de manière sécurisée les utilisateurs d'un site web ou les logs d'un réseau. En revanche, pour faire de l'analyse ponctuelle ou générer des graphiques à partir de milliers de lignes d'Open Data (comme des statistiques de ventes), un script de traitement de données (CSV/JSON) est souvent plus rapide et plus léger à mettre en œuvre.

## Comment

Pour l'approche "Base de données", j'ai utilisé des requêtes SQL structurées (avec des clauses `JOIN` pour lier les tables) afin d'extraire des informations précises sans surcharger la mémoire. Pour l'approche "Traitement de données", j'ai automatisé la récupération des CSV via des requêtes HTTP (module `requests` en Python), puis j'ai stocké temporairement ces données dans la mémoire vive sous forme de DataFrames (`pandas`) pour effectuer mes calculs d'agrégation (chiffre d'affaires par espèce).

## Mes difficultés

**L'intégrité et le formatage des données brutes.** Lors de l'exploitation des données de FranceAgriMer (SAÉ 1.05), j'ai réalisé que les fichiers plats (CSV) ne garantissent pas la propreté de l'information, contrairement à une base de données SQL où l'on peut imposer un type strict (ex: `INT` ou `FLOAT`). Le CSV contenait des nombres décimaux avec des virgules, ce qui bloquait mes calculs. J'ai dû coder moi-même des mécanismes de nettoyage pour remplacer ces caractères avant de pouvoir exploiter la donnée, une étape qui aurait été nativement gérée par un SGBD.

## Ce que j'en ai appris

J'ai compris qu'une donnée n'a de valeur que si elle est correctement structurée. J'ai appris à argumenter mes choix : je sais désormais que j'utiliserai une base SQL pour garantir l'intégrité et la sécurité (ex: mots de passe, gestion d'utilisateurs), mais que je privilégierai des bibliothèques comme `pandas` pour l'analyse statistique rapide de fichiers externes.

## Ce que je ferais autrement

Pour mes futurs projets de développement web (comme la création d'un back-end avec Flask), je ne ferai plus de requêtes SQL "brutes" directement dans mon code, car c'est source d'erreurs et de failles de sécurité. J'apprendrai à utiliser un ORM (*Object-Relational Mapping*), qui fera le pont automatiquement entre ma base de données SQL et mon code Python, rendant ainsi ma gestion de données beaucoup plus robuste et lisible.
